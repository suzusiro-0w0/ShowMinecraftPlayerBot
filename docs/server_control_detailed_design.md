# Discord向けMinecraftサーバー制御機能 詳細設計書

<!-- この文書は実装に必要な詳細設計をまとめたもの -->

## 目的
- 既存のDiscord BotにMinecraftサーバーの起動・停止・再起動機能を追加し、サーバー状況を専用チャンネルで常時表示する。
- コマンド実行結果やエラー通知を整理し、利用者と管理者双方に必要な情報を提供する。

## システム構成
- `bot/` 配下に機能単位でモジュールを分割し、責務を明確化する。
- 主な構成要素と役割は以下のとおり。
  - `bot/main.py` : Botのエントリポイント。設定読込とCog登録を行う。
  - `bot/config.py` : 設定値・永続化ファイルの読み込み／保存を担う。
  - `bot/status_message.py` : サーバー状況メッセージの生成・更新・自動削除ユーティリティを提供。
  - `bot/server_control.py` : `mcrcon`を用いたRCON通信およびOSコマンド経由でのサーバー制御を実装。
  - `bot/cogs/status_updater.py` : サーバー状態を定期ポーリングし、状況メッセージを更新する。
  - `bot/cogs/server_commands.py` : 起動／停止／再起動コマンドを提供し、確認フローや削除処理を担当。
  - `bot/utils/error_reporter.py` : 管理者向けエラーチャンネルへ通知する共通機能。
  - `data/status_message.json` : サーバー状況メッセージID等を保存する永続化ファイル。

## データフロー
1. Bot起動時に`bot/main.py`が`bot/config.py`から設定値を読み込む。
2. Cog登録後、`StatusUpdaterCog`が`StatusMessageManager`を介して状況メッセージを確保し、バックグラウンドタスクを開始。
3. バックグラウンドタスクは一定周期で`ServerController.get_status()`を呼び出し、取得した状態を`StatusMessageManager.update()`で反映。
4. ユーザーがコマンドを発行すると`ServerCommandsCog`が状態を確認し、必要に応じて確認UIを提示。
5. サーバー制御が実行されると`ServerController`がOSコマンドおよびRCONを使って操作し、結果を呼び出し元へ返す。
6. エラーが発生した場合は`ErrorReporter`が管理者チャンネルへ通知し、必要に応じて状況メッセージにも反映。

## サーバー状況メッセージ管理
- **チャンネル確保**: 設定値`status_channel_id`を基にチャンネルを取得。Bot起動時に既存メッセージIDを`data/status_message.json`から読み込み、存在確認を行う。
- **メッセージの存在確認**: 指定IDのメッセージが存在しない場合、チャンネル内の過去メッセージを削除し、新規に状況メッセージを作成。新しいメッセージIDは永続化ファイルに保存する。
- **表示フォーマット**:
  - タイトル: オンライン人数を「オンライン人数: n人」の形式で表示する。
  - 本文: ログイン中プレイヤー名のみを改行区切りで列挙し、不在時は「現在参加者はいません」を表示する。
  - フッター: Botバージョンや注意書きなど任意情報。
- **更新頻度**: デフォルト30秒間隔（設定で変更可能）。`asyncio.Lock`を用いてメッセージ更新の排他制御を行う。
- **付随メッセージの整理**: テキストコマンドの返信メッセージを編集して進捗・結果を伝え、状況チャンネルには状況メッセージが常に表示されるように保つ。

### 永続化ファイル仕様
- パス: `data/status_message.json`
- 形式: JSON
- サンプル:
  ```json
  {
    "status_message_id": 123456789012345678,
    "last_known_state": "stopped",
    "last_player_count": 0
  }
  ```
- 永続化ファイルは`StatusMessageStorage`クラスが読み書きし、整合性が取れない場合は再生成する。

## サーバー制御ロジック
### `ServerController`クラス
- **責務**: RCONを用いたサーバー状態取得、外部コマンドによるサーバープロセス制御。
- **初期化引数**: RCONホスト／ポート／パスワード、起動・再起動コマンドパス、コマンドタイムアウト時間。
- **状態取得**:
  - RCONへ接続し`list`コマンドを実行。
  - 応答を解析し、プレイヤー人数と一覧を取得。
  - 応答失敗時はステータスを`unknown`とし、再試行回数と例外情報を添えて返却。
- **起動処理**:
  - 呼び出し元が停止中であることを判定した後、起動スクリプトを`asyncio.create_subprocess_shell`で実行。
  - コマンドの成否を終了コードで判断し、成功時は期待状態を`running`として一時状態を解除。
  - Minecraftサーバーのようにプロセスが継続するコマンドでは`COMMAND_TIMEOUT`超過時にプロセスを終了させず、バックグラウンドでの継続を許容したうえでユーザーへ注意文を返す。
- **停止処理**:
  - RCON経由で`stop`を送信し、成功・失敗を`ServerActionResult`で返却。
  - 呼び出し元が必要に応じて確認ダイアログやメッセージ削除を担当。
- **再起動処理**:
  - 再起動コマンドが設定されている場合はそれを実行。
  - 未設定時は停止処理と起動処理を連結し、両方が成功した場合のみ`success=True`を返す。
- **戻り値**: 各操作は`ServerActionResult`データクラスを返し、成功可否・エラーメッセージ・詳細ログを含む。
- **エラーハンドリング**: 例外発生時は`ServerControlError`を送出し、呼び出し元が管理者通知と状況メッセージ更新を行う。

## コマンド設計
### コマンド一覧
| コマンド | 権限 | 概要 |
| --- | --- | --- |
| `!start_server` | 管理者ロール所持者 | サーバーが停止中であれば起動する |
| `!stop_server` | 管理者ロール所持者 | サーバーが起動中であれば停止する。プレイヤーがいる場合は確認必須 |
| `!restart_server` | 管理者ロール所持者 | サーバーが起動中であれば再起動する。プレイヤーがいる場合は確認必須 |

### コマンド共通フロー
1. コマンド受信後に実行ユーザーの権限を検証。
2. `ServerController.get_status()`で現在状態を確認し、実行可能か判定。
3. 実行前に状況メッセージへ操作予定を反映（`pending`表示）。
4. 実行中は状況メッセージに進捗を表示し、完了後に最終状態へ更新。
5. 応答メッセージは対象チャンネルへの返信として返し、進行状況や結果を都度編集して伝える。

### 確認ダイアログ
- プレイヤーが存在する場合は`discord.ui.View`を使用したボタン式確認を提示。
- タイムアウトは30秒。タイムアウト時はキャンセル扱いにし、その旨をエラーチャンネルへ通知。

## 状態更新タスク
- Cog起動時に`asyncio.Task`としてループを生成。Bot停止時に確実にキャンセルする。
- 取得したサーバー状態は構造体に詰め、変更有無に関わらず最終更新時刻を反映。
- エラー発生時は一定回数（例:3回）までは警告表示に留め、閾値を超えた場合は状態を`unknown`に切り替え。

## エラーログ・監視
- エラーチャンネルIDを設定で保持。
- `ErrorReporter`は例外情報、ユーザー、実行コマンド、スタックトレースをまとめたEmbedを生成して送信。
- 重要なイベント（起動完了、停止完了、再起動完了）は管理者チャンネルにも通知。
- 権限不足やHTTPエラーにより送信できなかった場合はDiscord API例外を捕捉し、ログへ詳細を記録する。

## 設定ファイル拡張
- `example_config.ini`に以下のセクションを追加予定。
  - `[discord]`: Botトークン、ステータスチャンネルID、エラーチャンネルID、管理者ロールID。
  - `[server]`: RCONホスト、ポート、パスワード、状態更新間隔。
  - `[commands]`: 起動・停止・再起動スクリプトパス、コマンドタイムアウト。
- コメントは全て日本語で記述し、ユーザーが編集しやすい形にする。

## ロギング方針
- Python標準`logging`を利用し、以下のログを残す。
  - Bot起動・停止。
  - サーバー操作開始／成功／失敗。
  - 状態取得エラー。
- ログレベルは設定ファイルで変更可能にする。

## テスト計画
- 単体テスト: `ServerController`の状態解析関数をモックしたRCONレスポンスで検証。
- 疎通テスト: DiscordテストサーバーとダミーMinecraftサーバーを用意し、実際のコマンド動作とメッセージ削除を確認。
- 例外テスト: RCON接続失敗、コマンドタイムアウト時にエラーチャンネルへ通知されるかを確認。
