# Discordサーバー管理ボット詳細設計

## 1. 背景と設計方針
- 既存機能はDiscordプレゼンスへのサーバー稼働状況とログイン者数の表示のみであり、サーバーの起動・停止・再起動は手動対応が必要。Windows + WinSW を前提としつつ、Linux 対応への拡張も考慮したモジュール分割が求められる。
- ボットは Discord サーバー内で以下 2 チャンネルを前提とする。
  1. **ステータス兼コマンドチャンネル**: サーバー状況メッセージとコマンド入力を同居させるチャンネル。ステータスメッセージ以外の投稿は自動削除の対象となり、履歴がクリーンに保たれる。利用者はこのチャンネルで `!start_server` などのコマンドを入力し、Bot も同チャンネルで応答する。
  2. **管理者アラートチャンネル**: エラーや手動介入が必要なログを送信するチャンネル。監視や調査を行う際の唯一のログ蓄積場所とし、ステータス兼コマンドチャンネルから削除された情報もここで追跡できるようにする。
- 将来の Linux 対応を見据えて、サーバー制御を抽象化し、プラットフォーム固有処理を切り替え可能にする。

## 2. ディレクトリ構成案
- `bot/` :
  - `main.py` …… Discord Bot の起動ポイント。設定読込と Cog 登録を行う。
  - `config.py` …… `config.ini` の読み込みとデータクラス化。
  - `logging.py` …… ログ設定を初期化。
- `server_control/` :
  - `base.py` …… 抽象クラス `ServerController` と `ServerState` 列挙体を定義。
  - `winsw_controller.py` …… Windows + WinSW 実装。`subprocess` で WinSW を操作。
  - `linux_controller.py`（将来追加） …… `systemctl` など Linux 固有の操作を実装。
- `discord_features/` :
  - `status_manager.py` …… ステータスメッセージの作成・更新・永続化・定期更新タスク。
  - `command_handlers.py` …… 起動 / 停止 / 再起動コマンドの Cog 実装。コマンド処理完了後の削除予約も担当。
  - `channel_cleaner.py` …… ステータス兼コマンドチャンネルを巡回し、ステータスメッセージ以外を削除するバックグラウンドタスクを管理。
  - `utils.py` …… 共通のメッセージ削除・アラート通知・確認 UI を提供。
  - `messages.py` …… メッセージテンプレート（警告文、エラー文）を集中管理。
- `data/` : 永続化ファイルを保存するディレクトリ。JSON ファイルとログのみを想定。

## 3. 設定ファイル (`config.ini`) の想定項目
| セクション | キー | 用途 |
| --- | --- | --- |
| `[discord]` | `bot_token` | Discord ボットトークン |
|  | `status_command_channel_id` | ステータス兼コマンドチャンネル ID |
|  | `admin_channel_id` | アラート送信用チャンネル ID |
|  | `delete_delay_seconds` | コマンド応答削除までの猶予秒数 |
| `[server]` | `platform` | `windows` / `linux` を指定。抽象クラスから適切な実装を選択 |
|  | `winsw_path` | `winsw.exe` の絶対パス |
|  | `service_name` | WinSW で管理しているサービス名 |
|  | `status_poll_interval` | サーバー状態のポーリング周期（秒） |
|  | `status_message_interval` | ステータスメッセージ定期更新の周期（秒） |
| `[persistence]` | `status_store_path` | `data/status_message.json` などの永続化ファイルパス |
| `[cleanup]` | `channel_cleanup_interval` | 巡回削除タスクの実行周期（秒） |
|  | `max_delete_per_cycle` | 1 回の巡回で削除する最大メッセージ数 |
|  | `delete_spacing_seconds` | メッセージ削除ごとの待機秒数（レート制限対策） |

## 4. ステータスメッセージ永続化設計
### 4.1 ファイル構造
- ファイル: `data/status_message.json`
- 形式: JSON オブジェクト 1 件
- 保存するキーと内容

| キー | 型 | 説明 |
| --- | --- | --- |
| `channel_id` | 数値 | ステータス兼コマンドチャンネルの ID。設定と矛盾がある場合は再生成を行う。 |
| `message_id` | 数値 | 現在アクティブなステータスメッセージの ID。存在しない場合は再作成。 |
| `updated_at` | ISO8601 文字列 | ボットが最後に内容更新を行った時刻。監視やデバッグ用。 |
| `state_snapshot` | オブジェクト | 直近のサーバー状態のキャッシュ。`{"state": "RUNNING", "player_count": 3}` のように保存。 |
| `cleanup_cursor` | 文字列 | ステータス兼コマンドチャンネル巡回削除で利用する最後に処理したメッセージ ID。未設定の場合は最新 100 件など固定数から走査。 |

### 4.2 読み書きフロー
1. `status_manager.StatusMessageStore`（仮称）がファイルを読み込む。存在しない場合は空の状態で初期化。
2. ファイルの `channel_id` と設定ファイルの値が異なる場合は、ステータスメッセージを再生成して JSON を上書き。
3. メッセージ取得 API:
   - キャッシュされた `message_id` で `channel.fetch_message()` を試行。
   - `discord.errors.NotFound` が返った場合は新規メッセージを作成し、ID を再保存。
4. 更新 API:
   - サーバー状態とプレイヤー数を取得し、テンプレート `messages.STATUS_TEMPLATE` へ流し込む。
   - Discord メッセージを `edit` で更新し、`updated_at` と `state_snapshot` を JSON へ保存。
   - I/O 例外が発生した場合は管理者チャンネルへエラーを送信し、必要であれば再試行キューへ登録。
5. 巡回削除 API:
   - `cleanup_cursor` をもとにステータス兼コマンドチャンネルの履歴を取得し、ステータスメッセージ以外の投稿を削除する。
   - 利用者が入力したコマンドメッセージと Bot の応答メッセージの双方を削除対象とし、削除完了後は `cleanup_cursor` を更新する。
   - Discord のレート制限に配慮し、1 回の巡回で削除する件数や待機時間を設定可能にする。

## 5. サーバー状態取得と抽象化
- `server_control.base.ServerState` 列挙体
  - `STOPPED`, `STARTING`, `RUNNING`, `STOPPING`, `RESTARTING`, `UNKNOWN`
- `ServerController` 抽象メソッド
  - `get_state() -> ServerState`
  - `start() -> None`
  - `stop(force: bool = False) -> None`
  - `restart(force: bool = False) -> None`
  - `list_players() -> list[str]`
- WinSW 実装 (`winsw_controller.py`)
  - `subprocess.run([winsw_path, service_name, "status"])` 等で状態取得。
  - プレイヤー一覧は既存の `mcstatus` もしくは RCON ラッパーで取得。将来的にプラグイン化しやすいように `player_provider` を差し替え可能にする。
  - コマンドは成功時に `returncode == 0` を確認。失敗時は例外 `ServerOperationError` を送出し、呼び出し側がリトライやアラート通知を行う。

## 6. ステータス更新タスクの設計
1. `main.py` 起動時に `status_manager.initialize_status_message()` を呼び、ステータスメッセージ ID を確定させる。
2. `asyncio.create_task()` で `status_manager.periodic_update_loop()` を常駐タスクとして起動し、ステータス表示の自動更新を行う。
3. `asyncio.create_task()` で `channel_cleaner.periodic_cleanup_loop()` を起動し、ステータス兼コマンドチャンネルから不要メッセージを間引く。
4. ステータス更新ループの詳細
   - 設定の `status_message_interval` 毎にサーバー状態を `ServerController.get_state()` で取得。
   - 取得できなかった場合（例外発生）は管理者チャンネルへ通知し、次周期までスリープ。
   - 正常時はテンプレートへ流し込み、メッセージと JSON を更新。
5. 巡回削除ループの詳細
   - 設定の `channel_cleanup_interval` 毎に履歴を取得し、ステータスメッセージ以外を削除する。
   - 1 回の巡回で削除する上限数や削除間隔（例: 1 件ごとに 1 秒待機）を設定から読み込む。
   - 削除対象には利用者のコマンドメッセージと Bot の応答、確認ボタンメッセージなど全てを含める。
6. Bot 終了時は両タスクにキャンセルを投げ、終了処理で JSON 保存を確実に行う。

## 7. コマンド挙動詳細
### 7.1 共通要件
- `discord.ext.commands.Cog` で `@commands.command()` を利用。
- コマンドはステータス兼コマンドチャンネル以外からの呼び出しを拒否し、エラーメッセージを返信して即削除する。拒否イベントは管理者チャンネルへも通知。
- コマンド投稿者のメッセージ、Bot の応答メッセージ、確認 UI（ボタン / リアクション誘導文）は `config.delete_delay_seconds` 後に削除予約し、削除結果をログに残す。
- コマンド処理中に発生したメッセージ削除失敗は巡回削除タスクが後処理する。失敗記録は管理者チャンネルへ送信。
- 例外処理は `CommandInvokeError` などをキャッチし、ユーザーには一般エラー文、詳細は管理者チャンネルへ送信。
- サーバー操作の前後で `status_manager` に状態を通知し、ステータスメッセージ更新と JSON 永続化を統一的に行う。

### 7.2 `!start_server`
1. `get_state()` が `STOPPED` 以外ならエラーメッセージ送信。`RUNNING` / `RESTARTING` / `STARTING` で文面を変え、利用者が送信したコマンドメッセージも削除予約する。
2. 正常に進める場合は利用者メッセージを `utils.delete_later()` で削除予約し、案内用の Bot メッセージも同様に予約する。
3. `status_manager.mark_state(ServerState.STARTING)` を呼び、ステータスメッセージを即時更新。
4. `ServerController.start()` を実行。例外時は `status_manager` を `UNKNOWN` に更新し、エラーをアラート。
5. 起動完了待ち
   - `status_poll_interval` 毎に `get_state()` を呼び、`RUNNING` になるまで待機（タイムアウトは 5 分など設定で制御）。
   - タイムアウト時は `stop()` を提案するメッセージを管理者へ送信し、利用者向けには別途通知。通知メッセージも削除予約。
6. 完了後、ユーザーへ成功メッセージ送信 → 一定時間後に削除。削除失敗は巡回タスクにより後始末。
7. ステータスメッセージを `RUNNING` 表示へ更新。

### 7.3 `!stop_server`
1. 状態確認。`STOPPED` / `STOPPING` / `RESTARTING` の場合はエラー返信。
2. `list_players()` でログイン者確認。
   - 0 人なら即停止へ進む。
   - 1 人以上の場合は `discord.ui.View` で「停止する / キャンセル」ボタンを表示。制限時間内に押されなければキャンセル。確認メッセージとボタンメッセージも削除予約する。
3. 停止処理
   - 承認後 `status_manager.mark_state(ServerState.STOPPING)` を呼ぶ。
   - `ServerController.stop()` を実行し、ポーリングで `STOPPED` まで待機。
   - タイムアウト時や例外発生時はアラートチャンネルへ送信し、ユーザーには失敗通知。
4. 停止完了後、成功メッセージを返信。プレイヤーがいた場合は一覧も添える。返信は削除予約し、巡回削除タスクと二重で管理。
5. ステータスメッセージを `停止中` 表示へ更新。停止に至らなかった場合は元の状態に復旧させる。

### 7.4 `!restart_server`
1. `RUNNING` 以外ならエラー返信。
2. プレイヤー確認 → 承認フローは停止と同じロジックを共有化。確認メッセージとボタンは削除予約し、巡回タスクでフォロー。
3. `status_manager.mark_state(ServerState.RESTARTING)` でステータスを切替。
4. `ServerController.restart()` を実行。内部では `stop()` → ポーリング → `start()` の順。WinSW に再起動コマンドがある場合は直接呼び出すオプションも提供。
5. 再起動完了後 `RUNNING` へ戻ったことを確認し、成功メッセージを返信。返信は削除予約。
6. ステータスメッセージを最終状態へ更新。失敗した場合は `UNKNOWN` に切り替え、管理者へ調査依頼を送る。

### 7.5 コマンドエラーハンドリング
- 各コマンドは共通デコレータで例外捕捉。
- `ServerOperationError` やネットワークエラー発生時
  - ユーザー: 「内部エラーが発生しました。管理者チャンネルをご確認ください。」
  - 管理者: 例外スタックトレース、入力コマンド、実行ユーザー、タイムスタンプを送信。
- ステータス兼コマンドチャンネル以外でコマンドを打たれた場合は「指定チャンネルで実行してください」と返信し即削除。拒否されたユーザー名とチャンネル情報を管理者チャンネルへ送る。
- 権限不足（ロールチェック）時は拒否し、誰が試したかを管理者チャンネルへ通知。拒否メッセージは削除予約し、巡回タスクでも削除。

## 8. メッセージテンプレート設計
- `messages.py` で定義するテンプレート例
  - `STATUS_TEMPLATE`: サーバー状態・プレイヤー数・プレイヤー一覧を整形。
  - `START_SUCCESS`, `START_FAIL`, `STOP_CONFIRM`, `STOP_SUCCESS`, `STOP_FAIL`, `RESTART_SUCCESS`, `RESTART_FAIL`。
  - `ERROR_UNKNOWN`, `ERROR_PERMISSION`, `ERROR_WRONG_CHANNEL`。
- 各テンプレートは `{server_name}`, `{state_label}`, `{player_count}`, `{players}` などのプレースホルダを持つ。
- 将来的に多言語化を想定し、メッセージキーで管理。

## 9. ログ・監視設計
- Python ログは `bot/logging.py` で `logging.config.dictConfig` を利用し、ファイル出力 + コンソール出力を設定。
- サーバー制御実行時・例外時には INFO / ERROR ログを残し、管理者チャンネルへの通知内容と整合。
- 重要な操作（停止 / 再起動）は監査目的で `data/audit.log` などへ追記する案も検討。
- チャンネル巡回削除の削除件数や失敗情報も INFO / WARNING として記録し、レート制限に達した際の調整材料とする。
